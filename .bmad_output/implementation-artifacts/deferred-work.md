## Deferred from: code review of story-4.5 (2026-03-30)

- **`upsert: false` gives no "already exists" signal** (`upload-reference.ts`, `upload-generated.ts`) — callers cannot distinguish duplicate-key errors from real storage failures. Intentional per spec; Story 2.5 must handle deduplication before calling `uploadReference`.
- **Signed URL TTL (3600s) too short for WhatsApp delivery retries** (`get-signed-url.ts`) — WhatsApp may retry delivery hours later; URL will be expired. Consider making TTL configurable or regenerating URLs on delivery attempt (Story 4.6 concern).
- **No filename/orderId sanitization** (`upload-reference.ts`, `upload-generated.ts`) — path injection possible if caller passes unvalidated input. Story 2.5 must validate/sanitize filenames and orderId before calling storage functions.
- **No retry logic on transient Supabase errors** — all five async operations throw immediately on first failure. Consider adding exponential backoff at the bot-engine layer (Story 2.5 / 4.5).
- **`uploadGenerated` hardcodes `image/png`** (`upload-generated.ts:10`) — if Gemini returns non-PNG bytes, stored MIME type is wrong. Story 4.3 should ensure AI output is always PNG or pass content type dynamically.
- **No file size validation** — WhatsApp photos can exceed Supabase free-tier 50 MB limit. Story 2.5 should validate size before calling `uploadReference`.
- **Concurrent uploads for same `orderId/filename`** — `upsert: false` causes one of N concurrent uploads to throw for identical paths. Story 2.5 must ensure unique filenames per photo (e.g., timestamp or UUID suffix).
- **`mock.module` hoisting fragility** (`*.test.ts`) — Bun's `mock.module` requires hoisting; test reliability depends on `bunfig.toml` preload. Monitor if Bun test runner behavior changes.

## Deferred from: code review of story-2.1 (2026-03-29)

- **In-memory state not suitable for serverless** (`bot.ts`) — `createMemoryState()` loses state across function invocations. MVP only. Story 2.2 should upgrade to Redis state adapter for production.
- **No 5-second timeout guard on webhook** (`route.ts`) — Chat SDK returns 200 immediately and processes async, but no explicit timeout enforcement at route layer. Monitor in production.
- **HMR double handler registration in dev** (`bot.ts`) — `onNewMention` could register twice during Next.js hot reload. Dev-only concern; no production impact.
- **Module-level Chat construction throws if env missing** (`bot.ts`) — `@t3-oss/env` validates at startup. Same pattern as storage package. Acceptable.

## Deferred from: code review of story-2.2 (2026-03-29)

- **Duplicate order race condition** (`bot.ts`) — Two concurrent webhooks for same sender can both create orders. Chat SDK queues per thread, but DB-level uniqueness constraint needed for full protection. Story 2.3+ should consider a unique constraint or locking.
- **AWAITING_FEEDBACK→REVISION_2 direct transition** (`state-machine.ts`) — Business logic must determine which revision number to use based on order history. AI Agent (Story 2.3) or revision handler (Story 5.2) must enforce this.
- **Stale active orders accumulate** (`conversation.ts`) — loadActiveOrder returns newest, ignoring older orphaned orders. Story 5.4 (abandoned cart) should add cleanup.
- **AWAITING_PAYMENT→GENERATING has no payment guard** (`state-machine.ts`) — Payment verification enforced by calling code in Story 3.x, not at state machine level.

## Deferred from: code review of story-4.6 (2026-03-30)

- **`imageUrl` path not validated before `getSignedUrl`** (`packages/bot-engine/src/deliver-image-to-client.ts`) — `imageUrl` is passed directly to `getSignedUrl` without verifying it matches the expected pattern `generated/{uuid}/{integer}.png`. The value originates from `uploadGenerated`'s return value (not user input), making injection unlikely; however a future refactor that plumbs the value differently could introduce risk. Add a regex guard at function entry in Epic 7 hardening.
- **Duplicate delivery messages on QStash retry after partial failure** (`packages/bot-engine/src/deliver-image-to-client.ts`) — If the warm message and photo are sent successfully but the document call fails, QStash retry will resend the warm message and photo to the client. Noted as MVP-acceptable in story spec. Mitigation options: track per-message delivery state in a `Generation` column, or use a WhatsApp media ID (upload once, reference multiple times). Defer to Epic 7 resilience story.
- **`clientName` not sanitized in `deliverImageToClient`** (`packages/bot-engine/src/deliver-image-to-client.ts`) — `clientName` is accepted as a parameter but unused in the warm message for MVP. If personalization is added in a future story, apply the same `sanitizeName` pattern from `send-payment-confirmation.ts` (strip `\r\n\t\x00-\x1F\x7F`) before embedding the name in any message body.
- **`deliver_image_invalid_recipient_phone` does not expose partial phone for debugging** (`packages/bot-engine/src/deliver-image-to-client.ts`) — The phone validation log omits the offending value to avoid logging PII. If debugging bad-phone production incidents is difficult, consider logging a truncated/masked form (e.g. first 4 digits + length) in Epic 8 LGPD-compliant logging story.
